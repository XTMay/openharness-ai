# Translation Guide

OpenHarness AI uses a lightweight file-based translation workflow for the early project stage.

## Current Languages

- English: `README.md`
- Simplified Chinese: `README.zh-CN.md`

## README Naming Convention

Use GitHub-friendly README filenames:

```text
README.md          # English, default repository homepage
README.zh-CN.md    # Simplified Chinese
README.ja.md       # Japanese
README.ko.md       # Korean
README.es.md       # Spanish
README.fr.md       # French
```

Each README should include a language switcher near the top.

Example:

```md
English | [简体中文](README.zh-CN.md) | [日本語](README.ja.md)
```

## Translation Rules

- Keep English as the source of truth for project positioning and feature status.
- Keep translated READMEs structurally aligned with `README.md`.
- Do not translate code commands, package names, file paths, or API names.
- Translate explanatory text, headings, and user-facing descriptions.
- When a feature is not implemented yet, keep future-tense wording clear.
- Update every translated README when the quickstart, roadmap, or supported versions change.

## When to Add More Languages

Add a new README translation when one of these is true:

- A contributor volunteers to maintain it.
- The project has visible users from that language community.
- The translation helps launch or share the project in a specific ecosystem.

Recommended next translations:

1. Japanese: `README.ja.md`
2. Korean: `README.ko.md`
3. Spanish: `README.es.md`
4. French: `README.fr.md`

## Future Documentation Site

When documentation grows beyond roughly 10 to 15 pages, move long-form docs to a documentation site with built-in i18n.

Good options:

- Docusaurus for a full project website and versioned docs.
- VitePress for lightweight technical documentation.
- Material for MkDocs for Python-friendly documentation.

Until then, file-based README translations are simpler and easier for contributors to maintain.
